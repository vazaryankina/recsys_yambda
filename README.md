# Исследование последовательных моделей рекомендаций и эффекта feedback loop на данных сервиса Яндекс Музыка

Репозиторий содержит код к моей курсовой работе по последовательным музыкальным рекомендациям. Основная задача - next-track prediction: по истории прослушиваний пользователя нужно предсказать следующий трек.

В работе я сравниваю несколько подходов разной сложности: простые baseline-модели, GRU baseline и SASRec. После выбора финальной SASRec-модели я отдельно анализирую, как различаются рекомендации при обучении на organic и non-organic взаимодействиях.

## Что сделано в работе

В проекте я последовательно:

* формирую reduced dataset из большого event-level датасета
* строю leave-one-out split для задачи next-track prediction
* сравниваю Most Popular, Last Item, Item-to-Item, GRU и SASRec
* провожу fine-tuning SASRec
* оцениваю модели не только по ranking-метрикам, но и по структуре рекомендаций
* отдельно сравниваю SASRec-модели, обученные на organic и non-organic историях
* дополнительно проверяю устойчивость результатов через bootstrap confidence intervals.

Для меня в этой работе важным оказывается не только то, попадает ли модель в следующий трек, но и то, какую выдачу она формирует: насколько широко использует каталог, сколько уникальных треков рекомендует и не замыкается ли на небольшом наборе популярных композиций.

## Структура репозитория

Код разбит на шесть основных ноутбуков. Их лучше смотреть по порядку, потому что результаты ранних этапов используются дальше.

### 1. [notebook_1_final_subsample.ipynb](notebook_1_final_subsample.ipynb)

В первом ноутбуке я формирую reduced dataset для дальнейших экспериментов.

Здесь загружаются исходные listening-события, строятся пользовательские признаки и отбирается финальная выборка. При отборе я учитываю активность пользователей, played ratio, долю organic-взаимодействий и распределение треков по popularity bins.

Итоговая выборка содержит:

* 10 206 636 событий
* 4 998 пользователей
* 300 000 треков.

После этого сохраняется файл final_events.parquet. Он используется в следующих ноутбуках, но не загружен в репозиторий из-за размера.

### 2. [notebook_2_eda_baselines.ipynb](notebook_2_eda_baselines.ipynb)

Во втором ноутбуке я провожу EDA финальной выборки, строю leave-one-out split и считаю первые модели.

Здесь рассматриваются:

* Most Popular;
* Last Item;
* Item-to-Item;
* GRU baseline.

Модели сравниваются по Recall@K, NDCG@K и MRR@K. Дополнительно я считаю Coverage@K, Unique@K и Gini@K, чтобы посмотреть не только на точность, но и на структуру выдачи.

Среди простых baseline-моделей лучше всего сработал Item-to-Item. Это важный результат, потому что он показывает, что локальные переходы между соседними треками являются сильным сигналом для музыкальных рекомендаций.

### 3. [notebook_3_sasrec.ipynb](notebook_3_sasrec.ipynb)

В третьем ноутбуке я перехожу к SASRec.

Здесь реализуется модель на PyTorch и проверяются первые конфигурации: базовая SASRec-архитектура, более сильная версия и вариант с negative sampling.

По результатам этих экспериментов full-softmax SASRec оказался сильнее sampled negative-варианта. Поэтому дальше я донастраиваю именно full-softmax SASRec.

### 4. [notebook_4_sasrec_final.ipynb](notebook_4_sasrec_final.ipynb)

В четвёртом ноутбуке проводится fine-tuning SASRec и выбирается финальная конфигурация.

Я проверяю несколько направлений улучшения:

* длину пользовательской истории
* число обучающих состояний на пользователя
* размер hidden representation
* параметры attention-блока
* длительность обучения.

Финальная модель выбирается по validation Recall@50, после чего один раз оценивается на test. Итоговая SASRec-модель получила Recall@50 = 0.2711 и дальше используется как основная модель для анализа feedback loop.

### 5. [notebook_5_feedback_loop.ipynb](notebook_5_feedback_loop.ipynb)

Пятый ноутбук посвящён feedback loop analysis.

Здесь я сравниваю две SASRec-модели с одинаковой архитектурой, но разными обучающими данными:

* SASRec organic-only - модель обучается только на organic interactions;
* SASRec nonorganic-only - модель обучается только на non-organic interactions.

Я сравниваю эти модели по ranking-метрикам, coverage, unique tracks, Gini, popularity bins и качеству на разных типах test targets.

Результат получился таким: organic-only модель показывает более высокое качество и шире покрывает каталог, а nonorganic-only модель сильнее концентрирует рекомендации вокруг популярных треков. Это не доказывает feedback loop причинно, но показывает, что тип обучающих взаимодействий влияет на поведение рекомендательной модели.

### 6. [notebook_6_bootstrap_confidence_intervals.ipynb](notebook_6_bootstrap_confidence_intervals.ipynb)

Шестой ноутбук представляет из себя дополнительную проверку устойчивости результатов из feedback loop analysis.

В нём я рассчитываю 95% доверительные интервалы для разницы между organic-only и nonorganic-only SASRec-моделями. Расчёт проводится по сохранённым рекомендациям из пятого ноутбука:

notebook_5_feedback_loop_outputs/recommendations.parquet

Для каждого пользователя используются target track, список рекомендаций и вариант модели. Затем считаются индивидуальные значения Recall@K, NDCG@K и MRR@K, после чего пользователи повторно выбираются с возвращением. В работе используется 5000 bootstrap-итераций и random seed 42.

Эта проверка нужна, чтобы убедиться, что преимущество organic-only модели по основным ranking-метрикам не является случайным результатом конкретной выборки пользователей.

## Данные

В работе используется Yambda — открытый event-level датасет музыкальных взаимодействий. Я работаю в основном с событиями прослушивания, потому что они напрямую связаны с задачей next-track prediction.

После первого ноутбука создаётся основной промежуточный файл final_events.parquet. Он дальше используется как вход для остальных этапов.

## Методология

Для оценки моделей используется leave-one-out split:

* последнее событие пользователя становится test target;
* предпоследнее событие становится validation target;
* все более ранние события образуют train history.

Такой split сохраняет порядок пользовательской истории. Это важно, потому что в задаче next-track prediction модель должна предсказывать следующий трек только по прошлым взаимодействиям.

Основной горизонт сравнения - K = 50. Часть метрик дополнительно считается при K = 10.

## Метрики

Я использую две группы метрик.

Первая группа оценивает качество ранжирования:

* Recall@K
* NDCG@K
* MRR@K.

Вторая группа описывает структуру рекомендаций:

* Coverage@K
* Unique@K
* Gini@K
* распределение рекомендаций по popularity bins.

Такой набор метрик нужен, потому что модель может хорошо попадать в следующий трек, но при этом рекомендовать слишком узкий и слишком популярный набор композиций.

## Основные результаты

1. Most Popular оказался самым слабым baseline, потому что почти не учитывает индивидуальную историю пользователя.

2. Last Item работает лучше, так как последний прослушанный трек сам по себе является сильным sequential-сигналом.

3. Item-to-Item стал лучшим среди простых baseline-моделей. Это показывает, что локальные переходы между треками очень важны для музыкальных последовательностей.

4. GRU baseline использует последовательность, но в простой конфигурации не смог обогнать Item-to-Item.

5. SASRec после fine-tuning показал лучший Recall@50 среди рассмотренных моделей и шире покрыл каталог.

6. При этом даже сильная SASRec-модель остаётся смещённой в сторону популярных треков. Поэтому качество рекомендаций нельзя оценивать только по ranking-метрикам.

7. В feedback loop analysis organic-only модель оказалась сильнее nonorganic-only модели по качеству и покрытию каталога. Nonorganic-only модель чаще концентрировалась вокруг популярных треков. Это показывает, что источник обучающих взаимодействий влияет не только на точность модели, но и на структуру рекомендаций.
